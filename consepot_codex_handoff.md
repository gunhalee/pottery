## 0. 문서 목적

이 문서는 Consepot 도자기 공방의 온라인 브랜드 사이트 개발을 위해 지금까지 논의한 결정사항, 구현 방향, 보류/쟁점, Codex가 이어서 확인해야 할 사항을 정리한 핸드오프 문서다.

이번 논의의 목적은 단순 채널 전략이 아니라 **브랜드 사이트 개발 요구사항 정의**다. 이후 Codex 세션에서는 이 문서를 기반으로 실제 저장소 구조, 라우팅, 컴포넌트, 카페24 연동 방식, 데이터 모델, 구현 순서를 구체화한다.

## 1. 비즈니스 컨텍스트

Consepot은 도자기 공방의 온라인화 프로젝트다. 온라인 사이트는 단순 쇼핑몰이 아니라 다음 역할을 동시에 수행해야 한다.

1. **브랜드 본진**
   - 공방의 철학, 제작 방식, 공간 분위기, 작업물의 감도 전달
   - 브랜드 아이덴티티와 아카이브 축적

2. **도자기 구매 경험**
   - 상시 상품과 1점물 판매
   - 가격 공개
   - 품절 작업물도 아카이브로 유지
   - 한정 상품은 `limited` 뱃지로 구분
   - 결제/주문/배송은 카페24를 기준 시스템으로 사용

3. **클래스 경험**
   - 원데이 클래스와 정규 클래스를 구분
   - 클래스 설명은 자체 사이트에서 충분히 제공
   - 예약은 네이버 예약으로 연결
   - 카카오채널 상담 예약은 보조적으로 사용

4. **커뮤니티/관계 형성**
   - 초기에는 로그인 기반 커뮤니티가 아니라, 작업물 아카이브·후기·소식·카카오채널 알림 중심
   - 고객 알림은 이메일이 아니라 카카오채널 중심
   - 구매 고객과 클래스 고객은 초기에 별개 고객군으로 간주하되, 사이트와 카카오채널을 통해 느슨하게 연결

## 3. 현재 사이트에 대한 참고 상태

현재 공개 사이트는 대략 다음 구조를 가진다.

```text
/
├─ About
├─ News
├─ Gallery
├─ Class
└─ Shop
```

현재 사이트에는 `Brand Name`, `Headline Text Here`, `hello@example.com` 등 placeholder 성격의 텍스트가 남아 있다. 개발 과정에서 실제 브랜드명, 카피, 연락처, 운영 정보, 정책 링크로 대체해야 한다.

현재 홈은 `Shop`, `Class`, `Gallery`, `News`로 진입하는 구조를 이미 갖고 있으므로, 큰 메뉴 방향은 유지 가능하다. 다만 실제 커머스/예약/알림 흐름은 아직 구현 전제로 보인다.

## 5. 카페24 연동 방향

### 5.1 결정된 원칙

- 가격은 카페24 기준으로 관리한다.
- 재고는 카페24 기준으로 관리한다.
- 상품 목록은 Consepot 사이트에서 직접 렌더링한다.
- 상품 목록을 카페24 웹 컴포넌트로 통째로 대체하지 않는다.
- Headless/API 연동은 당장 검토한다. 추후 과제가 아니다.
- 결제/주문 흐름은 카페24를 사용한다.

### 5.2 주문서 임베딩 이슈

논의 중 새로 나온 쟁점:

> 카페24 주문서를 브랜드 사이트 위에 임베딩할 수 있는가? 해도 되는가?

현재 기본 판단은 다음과 같다.

- 카페24 주문서를 iframe/modal로 Consepot 사이트 안에 임베딩하는 방식은 기술·보안·약관·결제 안정성 측면에서 신중하게 검토해야 한다.
- 공식적으로 지원되는 방식인지 먼저 확인해야 한다.
- 공식 지원이 불명확하면 iframe 임베딩은 피하고, 카페24 상품상세 또는 주문서로 이동시키는 방식이 안전하다.
- 이동감을 줄이기 위해 카페24 쪽 도메인을 `shop.consepot...` 형태의 서브도메인으로 맞추는 방안을 검토한다.

Codex 세션에서 확인할 것:

```text
1. 카페24 주문서 iframe 임베딩이 공식적으로 허용/지원되는지
2. 결제 페이지 임베딩 시 X-Frame-Options, CSP, SameSite cookie, PG/간편결제 이슈가 있는지
3. 카페24 상품상세 또는 주문서로 바로 이동하는 공식 URL 패턴이 있는지
4. 카페24 headless/API 사용 시 checkout redirect best practice가 무엇인지
```

### 5.3 가능한 구현 패턴

#### 옵션 A. Consepot 목록 → 카페24 상품상세 → 카페24 주문서

```text
/shop
↓
카페24 상품상세
↓
카페24 주문서
```

장점:
- 구현이 빠름
- 옵션/수량/결제 흐름을 카페24 기본 기능에 맡길 수 있음
- 결제 안정성이 높음

단점:
- 브랜드 경험이 카페24 화면으로 이동하면서 끊길 수 있음
- 품절 아카이브, 작업물 노트, Limited 뱃지 등 Consepot 고유 UX가 약해질 수 있음

#### 옵션 B. Consepot 목록 → Consepot 작업물상세 → 카페24 구매/주문서

```text
/shop
↓
/shop/[slug]
↓
구매 CTA
↓
카페24 상품상세 또는 주문서
```

장점:
- 브랜드 경험 유지
- 작업물 노트, 제작 과정, 관리법, 품절 아카이브, 클래스 연결이 자연스러움
- 도자기 공방의 감도와 콘텐츠를 직접 소유 가능

단점:
- 카페24 상품 데이터와 자체 상세 콘텐츠를 매핑해야 함
- 개발 범위가 커짐

현재 대화상에서는 사용자가 “상품 목록을 컴포넌트로 대체하기보다는 상세페이지가 카페24로 가도록”이라고 표현했으므로, Codex 세션에서 **상품 상세 페이지의 위치**를 반드시 재확인해야 한다.

해석 가능성:

```text
해석 1: Consepot 목록에서 클릭하면 카페24 상품상세로 이동한다.
해석 2: Consepot 안에 작업물 상세를 만들고, 구매 버튼만 카페24로 이동한다.
```

브랜드 사이트 관점의 추천은 옵션 B지만, 사용자의 표현은 옵션 A에 가까울 수 있으므로 첫 구현 전 명확히 결정해야 한다.

## 6. 상품 정책

### 6.1 상품 유형

초기에는 다음 상품 유형을 모두 고려한다.

| 유형 | 포함 여부 | 설명 |
|---|---:|---|
| 상시 상품 | 포함 | 반복 제작 가능하거나 재입고 가능한 생활 도자기 |
| 1점물 | 포함 | 재고 1개, 같은 형태/유약 재현이 어려운 작업물 |
| Limited | 포함 | 수량/기간/소성 회차/기획 한정 |
| 품절 작업물 | 포함 | 목록에 남기고 아카이브 자산으로 활용 |
| 커스텀 오더 | 제외 | 초기 범위에서 전부 제외 |
| 스마트스토어 전용 상품 | 제외 | 스마트스토어 초기 보류 |

### 6.2 가격 정책

- 모든 상품 가격은 공개한다.
- 가격은 카페24를 기준으로 관리한다.
- Consepot 사이트에 가격을 보여줄 경우 카페24 API 또는 동기화 데이터를 기준으로 한다.
- 가격을 하드코딩하지 않는 것을 원칙으로 한다.

### 6.3 품절 작업물 정책

- 품절 작업물도 목록에서 제거하지 않는다.
- 품절 작업물은 브랜드 아카이브이자 취향 형성 자료로 활용한다.
- 품절 카드에는 `품절` 뱃지를 표시한다.
- 품절 작업물에는 `재입고 요청` 또는 `비슷한 작업물 알림` 버튼을 제공한다.

권장 CTA 문구:

```text
상시 상품 품절: 재입고 알림 받기
1점물 품절: 비슷한 작업물 알림 받기
Limited 품절: 다음 한정 소식 받기
```

알림 CTA는 카카오채널로 연결한다.

### 6.4 Limited 뱃지

- Limited 상품은 목록과 상세에서 명확히 뱃지로 표시한다.
- Limited의 의미는 혼동되지 않게 구분해야 한다.

가능한 Limited 유형:

```text
수량 한정
기간 한정
소성 회차 한정
전시/프로젝트 한정
```

## 7. 상품 데이터 모델 초안

Codex 구현 시 아래 모델을 기준으로 검토한다.

```ts
type ProductKind = "regular" | "one_of_a_kind";

type AvailabilityStatus =
  | "available"
  | "sold_out"
  | "upcoming"
  | "archive";

type LimitedType =
  | "quantity"
  | "period"
  | "kiln_batch"
  | "project"
  | null;

type RestockCtaType =
  | "restock_alert"
  | "similar_work_alert"
  | "next_limited_alert"
  | null;

interface ConsepotProduct {
  id: string;
  slug: string;

  // Cafe24 mapping
  cafe24ProductNo: string;
  cafe24VariantCode?: string;
  cafe24ProductUrl?: string;
  cafe24CheckoutUrl?: string;

  // Display metadata
  titleKo: string;
  shortDescription: string;
  story?: string;
  category: "cup" | "plate" | "object" | "gift" | "limited" | "archive" | string;
  images: ProductImage[];

  // Product characteristics
  kind: ProductKind;
  isLimited: boolean;
  limitedType: LimitedType;
  isArchived: boolean;
  restockCtaType: RestockCtaType;

  // Commerce data, source of truth: Cafe24
  price: number;
  currency: "KRW";
  stockQuantity: number;
  availabilityStatus: AvailabilityStatus;

  // Ceramic-specific information
  material?: string;
  glaze?: string;
  size?: string;
  usageNote?: string;
  careNote?: string;
  shippingNote?: string;

  // Publishing
  published: boolean;
  publishedAt?: string;
  createdAt: string;
  updatedAt: string;
}

interface ProductImage {
  src: string;
  alt: string;
  isPrimary?: boolean;
}
```

데이터 기준:

| 데이터 | 기준 시스템 |
|---|---|
| 상품 번호 | 카페24 |
| 가격 | 카페24 |
| 재고 | 카페24 |
| 판매 가능 여부 | 카페24 |
| 옵션 | 카페24 |
| 작업물명 | 카페24와 Consepot 간 매핑 필요 |
| 스토리/작가 노트 | Consepot |
| 이미지 | 카페24 + Consepot, 운영 방식 결정 필요 |
| Limited 여부 | Consepot 또는 카페24 상품 태그 매핑 |
| 1점물 여부 | Consepot 또는 카페24 상품 태그 매핑 |
| 품절 아카이브 노출 여부 | Consepot |
| 재입고/비슷한 작업물 알림 CTA | Consepot |

## 8. Shop UX 요구사항

### 8.1 목록 페이지

경로 후보:

```text
/shop
```

목록에 포함할 것:

- 판매중 상품
- 품절 상품
- 상시 상품
- 1점물
- Limited 상품
- Archive 성격의 과거 작업물

필터 후보:

```text
전체
판매중
한정
상시 상품
단 하나의 작업물
품절/아카이브
```

정렬 후보:

```text
최신순
판매중 먼저
한정 먼저
가격 낮은순
가격 높은순
```

단, 초기에는 정렬을 최소화하고 필터만 제공해도 된다.

### 8.2 상품 카드

상품 카드 표시 요소:

```text
대표 이미지
뱃지: 판매중 / 품절 / 한정 / 단 하나의 작업물
작업물명
짧은 설명
가격
CTA
```

CTA 예시:

```text
판매중: 자세히 보기 / 구매하기
품절 상시 상품: 재입고 알림
품절 1점물: 비슷한 작업물 알림
Limited 품절: 다음 한정 소식 받기
```

### 8.3 상품 상세 페이지

상품 상세 필요 여부는 확정. 단, 위치는 추가 결정 필요.

Consepot 내부 상세를 만들 경우 경로:

```text
/shop/[slug]
```


주의:

## 11. 고객군과 후속 제안

### 11.1 고객군 구분

구매 고객과 클래스 고객은 초기에는 별개의 고객군으로 간주한다.

| 고객군 | 주요 관심사 | 주요 채널 |
|---|---|---|
| 구매 고객 | 신작, 입고, 품절/재입고, 관리법, 한정 드롭 | Consepot 사이트 + 카카오채널 |
| 클래스 고객 | 예약, 준비물, 수업 안내, 작업물 완성, 픽업, 다음 수업 | Consepot 사이트 + 네이버 예약 + 카카오채널 |

### 11.2 연결 원칙

별개의 고객군으로 보되, 완전히 분리하지는 않는다.

원칙:

```text
데이터와 메시지는 분리한다.
경험과 콘텐츠는 연결한다.
```

구매 고객에게는 처음부터 클래스를 강하게 판매하지 않는다. 대신 다음 콘텐츠로 자연스럽게 연결한다.

```text
도자기 관리법
작업물 제작 이야기
비슷한 작업물 아카이브
다음 소성 소식
한정 드롭 알림
```

클래스 고객에게는 처음부터 상품 구매를 강하게 판매하지 않는다. 대신 다음 콘텐츠로 자연스럽게 연결한다.

```text
수강생 작품 아카이브
완성/픽업 안내
다음 클래스
정규 클래스
공방 제작 작업물 둘러보기
```

### 11.3 카카오채널 활용

초기 고객 알림은 카카오채널 중심이다.

카카오채널 CTA 문구 후보:

```text
신작 소식 받기
다음 제작 소식 받기
재입고 알림 받기
비슷한 작업물 알림 받기
클래스 오픈 알림 받기
수업 문의하기
```

가능하면 CTA 링크에 URL 파라미터를 붙여 유입 맥락을 구분한다.

예시:

```text
?source=product_soldout&product=moon-jar-white
?source=class_oneday
?source=limited_drop
```

실제 카카오채널에서 파라미터를 어떻게 활용할지는 별도 확인 필요.

## 17. 외부 연동 및 환경 변수 후보

```env
CAFE24_MALL_ID=
CAFE24_CLIENT_ID=
CAFE24_CLIENT_SECRET=
CAFE24_ACCESS_TOKEN=
CAFE24_REFRESH_TOKEN=
CAFE24_API_BASE_URL=

NEXT_PUBLIC_NAVER_RESERVATION_ONEDAY_URL=
NEXT_PUBLIC_NAVER_RESERVATION_REGULAR_URL=
```

주의:

- Client secret, access token, refresh token은 절대 프론트에 노출하지 않는다.
- 카페24 API 호출은 서버 사이드, API route, build-time sync, 또는 별도 integration layer에서 처리한다.
- 토큰 갱신 방식과 권한 범위는 Codex가 실제 문서 확인 후 설계해야 한다.

- 상품 상세를 카페24 외부 페이지로 바로 보낼 경우, Consepot 사이트의 SEO 자산은 약해질 수 있다.

## 21. Codex에게 전달할 실행 프롬프트 초안

아래 내용을 Codex 세션 시작 시 그대로 전달하면 좋다.

```text
이 저장소는 Consepot 도자기 공방의 브랜드 사이트입니다.
목표는 카페24를 전체 사이트로 대체하는 것이 아니라, 현재 브랜드 사이트를 유지하면서 카페24를 상품/가격/재고/결제/주문 엔진으로 연동하는 것입니다.

주요 결정사항:
- 사이트는 한국어만 사용합니다.
- 현재 톤을 유지합니다.
- 스마트스토어, 검색광고, 이메일 뉴스레터, 커스텀 오더는 초기 범위에서 제외합니다.
- 상품 목록은 자체 사이트에서 직접 구현합니다.
- 가격과 재고는 카페24 기준입니다.
- 상시 상품과 1점물 모두 판매합니다.
- 품절 작업물도 목록에 남기고, 재입고/비슷한 작업물 알림 CTA를 둡니다.
- Limited 상품은 뱃지로 표시합니다.
- 클래스는 원데이/정규로 나누고 네이버 예약으로 연결합니다.
- 상담/알림은 카카오채널 중심입니다.

우선 검토할 것:
1. 현재 프로젝트 구조와 라우팅을 파악하세요.
2. 상품 상세 페이지를 Consepot 내부에 둘지, 카페24 상품상세로 보낼지 선택지별 구현 영향을 정리하세요.
```

## 23. 최종 현재 상태 요약

현재까지의 잠정 결론은 다음과 같다.

> Consepot 사이트는 브랜드 본진으로 유지한다. 카페24는 상품·가격·재고·결제·주문·배송의 기준 시스템으로 사용한다. 상품 목록은 Consepot 사이트에서 직접 구현하고, 품절 작업물까지 포함해 브랜드 아카이브로 유지한다. 상시 상품과 1점물을 모두 다루며, Limited 상품은 뱃지로 구분한다. 스마트스토어는 초기에는 보류한다. 클래스는 원데이와 정규로 나누고 네이버 예약으로 연결한다. 상담과 알림은 카카오채널 중심으로 운영한다. 텀블벅은 프로젝트 채널로만 활용한다. 가장 먼저 결정해야 할 개발 쟁점은 상품 상세 페이지의 위치와 카페24 주문서 처리 방식이다.
